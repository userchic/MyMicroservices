# MyMicroservices
Примитивный проект попытка сделать микросервисное приложение.  Судя по всему законы сильно подавляют развитие свободных соцсетей поэтому проект не будет развернут, хотя он не будет развернут поскольку является способом получить практику по микросервисам, чтобы можно было это предъявить где нибудь в резюме - при поиске работы.
# Суть проекта

Сделать простенькую соц сеть или что-то на подобие того где можно делать посты, редактировать и удалять. Возможно будет сделана подборка постов. Также есть сервис аутентификации который будет выдавать JWT токен с идентификатором пользователя чтобы другие сервис(ы) могли делать объекты со ссылкой на пользователя. 

## Состояние
На данный момент сделаны:
- созданы 5 микросервисов на базе единственных сущностей: 
   - сервис аутентификации предоставляющий соответствующие CRU возможности и возможности по поиску пользователей кэширование запросов отдельных профилей. API документация генерируется и организовано логирование; Написаны тесты сервиса и валидации,
   - сервис постов позволяющий их создавать(сообщение отправляется в Kafka), изменять и удалять. API документация генерируется и организовано логирование. Написаны тесты;
   - сервис подписок с возможностью подписываться и отписываться от пользователей. API документация генерируется и организовано логирование; Написаны тесты сервиса и валидации;
   - сервис уведомлений позволяющий устанавливать правила уведомлений и играющий роль Consumer-а потребляющего сообщения о публикации постов и отправляющего уведомления подписчикам по SMTP протоколу.API документация генерируется и организовано логирование; Написаны тесты сервиса и валидации;
   - сервис комментариев для работы с комментариями под постами. API документация генерируется и организовано логирование; Написаны тесты сервиса и валидации;
- сделан GateWay с помощью YARP
- аутентификация по JWT. JWT призван предоставлять сервисам в payload номер пользователя. 
- сделан фронтенд на React. Реализована работа с постами, подписками, правилами уведомлений и аутентификация с регистрацией. Также реализован поиск пользователей. 
- создан docker-compose файл способный развернуть систему
- в принципе можно масштабировать, только придется со временем ввести паттерны масштабирования БД если поставить задачу обрабатывать много данных

## Масштабируемость

В связи с прогрессом в понимании Shared DB и Database Per Service я так понял, что система масштабируема больше чем я себе представлял и конечно тут есть очередь Kafka.
И эта самая очередь не обладает специальным партиционированием, сообщения просто обрабатываются консюмерами из партиций. Эти самые партиции могут распределить нагрузку по консюмерам, хотя это можно сделать еще и организовав консюмеров по собственному видению - распределить между ними партиции определенным образом - но по БД нагрузка должна будет распределена отдельно.

Консюмеры запрашивают подписчиков из соответствующего сервиса. Это по идее представляет из себя большой спрос на чтение и информацию можно реплицировать. Возможно изменений в подписках будет не слишком много, но вообще мне кажется что при необходимости можно применить все возможное из списка: репликация, партиционирование, шардирование - чтобы распределить нагрузку и запросы на чтение в том или ином проекте.
