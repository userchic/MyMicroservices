# MyMicroservices
Примитивный проект попытка сделать микросервисное приложение.  Судя по всему законы сильно подавляют развитие свободных соцсетей поэтому проект не будет развернут, хотя он не будет развернут поскольку является способом получить практику по микросервисам, чтобы можно было это предъявить где нибудь в резюме - поиске работы.
# Суть проекта

Сделать простенькую соц сеть или что-то на подобие того где можно делать посты, редактировать и удалять. Возможно будет сделана подборка постов. Также есть сервис аутентификации который будет выдавать JWT токен с идентификатором пользователя чтобы другие сервис(ы) могли делать объекты со ссылкой на пользователя. Фронтенд работает с аутентификацией и после нее спрашивает с других сервисов - что у них есть.

## Состояние
На данный момент сделаны:
- Созданы 4 микросервиса на базе единственных сущностей: Сервис аутентификации предоставляющий соответствующие CRU возможности и возможности по поиску пользователей+кэширование запросов отдельных профилей; Сервис постов позволяющий их создавать(сообщение отправляется в Kafka), изменять и удалять; Сервис подписок с возможностью подписываться и отписываться от пользователей; Сервис уведомлений позволяющий устанавливать правила уведомлений и играющий роль Consumer-а потребляющего сообщения о публикации постов и отправляющего уведомления подписчикам по SMTP протоколу.
- Сделан GateWay с помощью YARP
- аутентификация по JWT. JWT призван предоставлять сервису постов в payload номер пользователя. 
- сделан фронтенд на React. Реализована отправка постов, редактирование и удаление и аутентификация с регистрацией. Также реализован поиск пользователей. Можно подписываться на пользователей. можно работать с правилами уведомлений.
- В принципе можно масштабировать, система как бы соответствует паттерну Shared DB, но это антипаттерн так что масштабирование быстро подавит СУБД. Масштабировать можно все сервисы.
- Создан docker-compose файл для развертывания kafka в контейнере.

## Последние изменения

- Добавлен сервис уведомлений: работа с правилами уведомлений и одновременно с этим Consumer сообщений об отправке постов. Сервис запрашивает список подписчиков и описание пользователя опубликовавшего пост у соответствующих сервисов. 
- Уведомления отправляются с помощью SMTP протокола письмом на почту. Другие сервисы-способы уведомлений пока что не добавлены
- Добавлено поле Email к пользователю(UserService) - для контактов. Оно не переиспользуется сервисом уведомлений!

Подобранная до этого конфигурация Kafka оказалась не рабочей. На данный момент я смог подобрать другую с которой добился большей стабильности в доступности kafka. но kafka-ui не работает.

## Планы

- Будут уведомления о постах подписчикам. Это будет реализовано в первую очередь на основе очереди Kafka. Сообщения будут содержать идентификатор пользователя опубликовавшего пост. 
- Очередь будут читать консюмеры. Эти консюмеры будут запрашивать у сервисов список подписчиков автора и информацию об авторе чтобы уведомление содержало эту информацию.
- Предыдущий пункт приведет к спросу на сервис подписчиков. БД этого сервиса можно реплицировать для компенсации этого. Вряд ли эта БД будет пользоваться большим спросом на изменение данных. По большей части чтение.
- Получается что консюмер будет орудовать большими объемами данных(списки подписчиков) и распределив сообщения по партициям можно будет распределить спрос на сервис уведомлений.
- Сервис уведомлений будет отправлять уведомления по указанному пользователем сервису-инструменту-способу. Их надо будет запрограмировать, возможно применить паттерн Strategy.
- Сервис уведомлений для получения информации о подписчиках и постящих будет использовать HttpClient что обозначает что надо придумать пул клиентов чтобы не потратить память и слоты(кажется каждый клиент в своей работе занимает физическое устройство или его слот для отправки сообщений)

## Масштабируемость

Пока что для нее особых вещей сделано не было кроме Kafka в виде одной очереди. 
И эта самая очередь не обладает специальным партиционированием, сообщения просто обрабатываются консюмерами из партиций. Эти самые партиции могут распределить нагрузку по консюмерам, хотя это можно сделать еще и организовав консюмеров по собственному видению - распределить между ними партиции определенным образом - но по БД нагрузка должна будет распределена отдельно.

Консюмеры запрашивают подписчиков из соответствующего сервиса. Это по идее представляет из себя большой спрос на чтение и информацию можно реплицировать. Возможно изменений в подписках будет не слишком много, но вообще мне кажется что при необходимости можно применить все возможное из списка: репликация, партиционирование, шардирование - чтобы распределить нагрузку и запросы на чтение в том или ином проекте.
